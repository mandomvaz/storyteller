## Context

Stories are generated by the pipeline but the `Story` object is discarded after `VoiceStoryService.GenerateStoryFromTextAsync` returns — only paragraphs are pushed through `Channel<TextUnit>` for TTS. There is zero persistence.

The app runs on a Docker-based minipc (low resources). All state must be local file-based. The badge prompt is read from a file on startup; to change it, restart the container.

## Goals / Non-Goals

**Goals:**
- Persist completed stories to SQLite on user confirmation
- Generate 3-emoji badge via SK with prompt from configurable file
- Cache story in memory (10min TTL) while awaiting user decision
- Expose list (id + badge) and get-by-id for persisted stories
- Everything configurable via `appsettings.json` + env vars for Docker

**Non-Goals:**
- Auto-save or "guardar por defecto" — user must explicitly confirm
- Frontend save UI — only backend endpoints and services
- Editing or deleting stories
- Badge prompt live-reload — prompt is read once at startup; changing it requires container restart

## Decisions

### SQLite raw vs EF Core/Dapper
**Decision**: `Microsoft.Data.Sqlite` (raw ADO.NET)
**Rationale**: Only 4 operations (init, save, list, get). EF Core or Dapper add unnecessary weight for this scope. Raw SQLite keeps the dependency graph flat and the Docker image small.
**Alternatives considered**: EF Core (overkill), Dapper (still an extra dep for marginal gain).

### One table with JSON paragraphs vs two tables
**Decision**: Single `Stories` table, `Paragraphs` as `TEXT` JSON array.
**Rationale**: Paragraphs are always loaded/saved atomically with the Story. Never queried individually. Max ~8 paragraphs. Serialization with `System.Text.Json` is trivial.
**Alternatives considered**: Two tables (Stories + Paragraphs) — normalized but adds JOINs and more code for zero query benefit today.

### IMemoryCache vs ConcurrentDictionary for temp storage
**Decision**: `IMemoryCache` (built-in ASP.NET Core).
**Rationale**: Built-in, supports TTL, thread-safe, no extra deps. `ConcurrentDictionary` would need manual expiration logic.
**Alternatives considered**: `ConcurrentDictionary` + timer (more code, reinventing the wheel).

### KernelFunction injected via DI vs created per-request
**Decision**: Created once at startup from prompt file, registered as singleton.
**Rationale**: The `KernelFunction` is immutable. The prompt is read at startup; if it needs to change, restart the container. No hot-reload complexity needed.
**Alternatives considered**: Hot-reload with polling or FileSystemWatcher (unnecessary complexity for rare changes).

### Story id = JobId (same Guid)
**Decision**: The `Story.Id` reuses the `PipelineJob.JobId` Guid.
**Rationale**: The frontend already receives `jobId` from `POST /api/stories/new`. No need to generate or communicate a second id. The pipeline runner has access to both.

### PersistenceService as a separate service
**Decision**: New `PersistenceService` class, not merged into `IStoryRepository`.
**Rationale**: SRP — repository handles SQLite, persistence service handles the cache-then-persist orchestration. The service is the "two-phase" coordinator.

## Dependency Injection Registration

```csharp
// Program.cs additions:

// SQLite repository
builder.Services.AddSingleton<IStoryRepository, SqliteStoryRepository>();

// Memory cache for pending saves
builder.Services.AddMemoryCache();

// Persistence service
builder.Services.AddSingleton<PersistenceService>();

// Badge SK function — create from prompt file, register in DI
var promptPath = builder.Configuration["BadgePrompt:Path"] ?? "Prompts/badge.txt";
var prompt = File.Exists(promptPath) ? File.ReadAllText(promptPath) : DefaultBadgePrompt;
var badgeFunc = kernel.CreateFunctionFromPrompt(prompt, new OpenAIPromptExecutionSettings { MaxTokens = 10 });
builder.Services.AddKeyedSingleton<KernelFunction>("badge", badgeFunc);

// Badge service
builder.Services.AddSingleton<BadgeService>();
```

## PipelineRunner Changes

After `storyTask` completes, the runner captures the `Story` and calls badge + cache:

```csharp
// After storyTask completes (inside the WhenAny loop or after it):
if (storyTask.IsCompletedSuccessfully)
{
    var story = await storyTask;
    
    try 
    { 
        story.Badge = await _badgeService.GenerateBadgeAsync(story); 
    }
    catch { /* log, badge stays empty */ }
    
    await _persistenceService.CacheAsync(story);
}
```

This runs concurrently with TTS/delivery — the three tasks are already concurrent via `Task.WhenAny`. The badge/cache logic triggers as soon as `storyTask` finishes, regardless of TTS/delivery state.

## Risks / Trade-offs

- **[Risk] Cache expires before user saves** → 404 on save endpoint. User must re-record. Mitigation: 10min TTL is generous for a single-sitting interaction. Consider extending if needed.
- **[Risk] Badge generation fails silently** → Story cached with empty badge. Mitigation: log the error, badge remains empty string. The story is still persisted with no badge.
- **[Risk] SQLite concurrent access** — Currently only one pipeline runs at a time (single reader channel). If that changes, SQLite handles concurrent reads fine, writes use WAL mode. Monitor if needed.
